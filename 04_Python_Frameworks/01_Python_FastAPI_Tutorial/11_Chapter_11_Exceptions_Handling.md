# Error Handling

Error handling is a critical part of API design because it defines how the system communicates failures to clients. A well-designed error handling strategy ensures that errors are:

* Predictable
* Structured
* Meaningful
* Consistent

In FastAPI, error handling is built on top of Starlette’s exception system and integrates tightly with request validation and response generation.

At a system level, errors can occur during:

* Request parsing
* Validation
* Dependency resolution
* Business logic execution
* Response generation

FastAPI intercepts these failures and converts them into structured HTTP responses.

---

# Custom Exceptions

Custom exceptions allow you to define application-specific error conditions and return meaningful responses instead of generic failures.

---

# Using `HTTPException`

The simplest way to raise an error in FastAPI is with `HTTPException`.

## Example

```python id="m8c2x1"
from fastapi import FastAPI, HTTPException

app = FastAPI()

@app.get("/users/{user_id}")
def get_user(user_id: int):
    if user_id != 1:
        raise HTTPException(
            status_code=404,
            detail="User not found"
        )

    return {"user_id": user_id}
```

---

## Internal Flow

When this exception is raised:

1. Route execution stops immediately
2. FastAPI catches the exception
3. Converts it into a JSON response
4. Attaches the specified status code
5. Sends the response to the client

---

## Response Format

```json id="z1ow76"
{
  "detail": "User not found"
}
```

---

# Creating Custom Exception Classes

You can also define your own exception types using Python classes.

## Example

```python id="pk8s7f"
class UserNotFoundException(Exception):
    def __init__(self, user_id: int):
        self.user_id = user_id
```

---

## Why Use Custom Exceptions?

Custom exceptions help:

* Represent domain-specific failures
* Separate business errors from framework errors
* Improve readability
* Centralize handling logic

---

# Global Exception Handlers

Global exception handlers allow centralized error processing across the application.

Instead of repeating error formatting inside routes, you define it once globally.

---

# Creating a Global Exception Handler

## Example

```python id="l9f2tx"
from fastapi import Request
from fastapi.responses import JSONResponse

@app.exception_handler(UserNotFoundException)
async def user_not_found_handler(
    request: Request,
    exc: UserNotFoundException
):
    return JSONResponse(
        status_code=404,
        content={
            "error": f"User {exc.user_id} not found"
        }
    )
```

# Validation Errors

Validation errors occur when incoming request data does not match the expected schema.

These errors are automatically generated by FastAPI using Pydantic.

---

# Example Validation Setup

```python id="2cw5ap"

class User(BaseModel):
    name: str
```

# Customizing Validation Errors

You can override the default validation error handler.

## Example

```python id="c0n0jr"
from fastapi.exceptions import RequestValidationError

@app.exception_handler(RequestValidationError)
async def validation_exception_handler(
    request: Request,
    exc: RequestValidationError
):
    return JSONResponse(
        status_code=400,
        content={
            "errors": exc.errors()
        }
    )
```

---

# Why Customize Validation Errors?

Custom validation handling allows you to:

* Change status codes
* Match company API standards
* Simplify error responses
* Improve frontend integration

# Raising Errors in Dependencies

Errors can also occur during dependency resolution.

---

# Example

```python id="q2x9ea"
from fastapi import Depends

def verify_token(token: str):
    if token != "secret":
        raise HTTPException(
            status_code=401,
            detail="Invalid token"
        )

    return token

@app.get("/secure")
def secure_route(
    token: str = Depends(verify_token)
):
    return {"message": "Access granted"}
```

---

# Internal Dependency Error Flow

```text id="m9w2je"
Dependency Executes
        ↓
Validation/Authentication Fails
        ↓
Exception Raised
        ↓
Route Handler Skipped
        ↓
Error Response Returned
```

Dependencies execute before the route handler itself.

---

# Error Handling in Middleware

Middleware can also intercept unexpected errors globally.

## Example

```python id="4yot7m"
@app.middleware("http")
async def error_logging_middleware(
    request: Request,
    call_next
):
    try:
        response = await call_next(request)
        return response

    except Exception as e:
        print(f"Error occurred: {e}")
        raise e
```

---

# Middleware Error Handling Use Cases

* Logging unexpected failures
* Monitoring application health
* Sending alerts
* Tracking crash reports
* Adding correlation IDs

# Best Practices for Error Handling

## Use Appropriate Status Codes

Examples:

| Status Code | Meaning               |
| ----------- | --------------------- |
| `400`       | Bad Request           |
| `401`       | Unauthorized          |
| `403`       | Forbidden             |
| `404`       | Resource Not Found    |
| `422`       | Validation Error      |
| `500`       | Internal Server Error |

---

## Avoid Exposing Internal Details

Do not expose:

* Stack traces
* Database errors
* Internal architecture
* Sensitive system data

---

## Centralize Error Formatting

Use global exception handlers for consistency.

---

## Log Unexpected Errors

Unhandled exceptions should always be logged.

---

## Keep Error Messages Clear

Good:

```text id="4yw8s9"
User not found
```

Bad:

```text id="t2x5qm"
Database query failed unexpectedly due to internal lookup issue
```
